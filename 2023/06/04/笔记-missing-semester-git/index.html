笔记-missing-semester-git

介绍

从底层的数据结构介绍Git的工作原理，分析.git文件夹存储的内容的含义
git的提交历史可以看为一个有向无环图

最左侧为起始提交，从左往右的顺序看，除了第一个提交，其他提交都有一个父提交

底层数据结构

Git将文件目录称为tree，将文件称为blob

blob

blob就是对应文件的二进制数组，每次提交中的每个文件都会对应一个blob

tree

tree就是文件夹，父文件夹和子文件夹都是tree，通过指针的形式将文件夹连接起来，形式与文件系统中的目录树一致

可以将blob视为叶子节点，tree可以是非叶子节点或者叶子节点

在数据结构中，将tree表示为一个map，其中的内容是它的子节点或者叶子节点

commit

commit就是提交记录，每次执行git commit都会生成一个commit的对象

snapshot就是对于当前提交的一个快照，当前提交情况下的整个目录及文件的情况，所以是tree类型的，表示当前工作目录的文件树

一个反直觉的事情：git并没有直接记录每个文件的修改历史，而是会记录每个commit的所有文件的内容即blob，如果某个文件在多个commit中都未修改，这些commit的tree会使用同一份blob

git通过比较同一个文件的多个版本的blob的完整内容来获得修改记录，而不是通过修改记录来从旧版本文件推理得到新文件

objects

object是git中blob、tree、commit的统称

所有的object被当成一个map来存储，使用object的哈希值作为键

references

对于用户而言，使用hash来访问object的方式过于困难，git提供了reference的方式，来访问commit的hash

比如常用的master, HEAD等都是reference

Repository

可以简单的将git仓库看作是objects和reference的组合

Staging area

暂存区，将指定的修改加入到暂存区，然后commit，这个commit的snapshot就会只包含对该文件的修改后的内容的记录

Git目录

一个空的初始化的git仓库目录为：

HEAD：指的是当前所在分支的符号引用（symbolic reference），它指示当前工作树（working tree）所在的位置。
branchs ：这个文件夹存储了本地分支的引用。每个分支都有一个相应的文件，其中包含指向分支最新提交的引用。

config：这个文件保存了仓库的配置信息，包括用户名、邮箱、远程仓库的URL等。可以使用git config命令来修改和查看这些配置。

description：这是一个可选文件，用于描述仓库的作用和内容。当使用gitweb等工具浏览仓库时，该文件的内容可能会显示在仓库的描述信息中。

hooks：这个文件夹包含一些示例钩子脚本（hook scripts）。Git钩子是在特定的Git操作（如提交、推送等）发生时自动执行的脚本。

info/exclude：这是一个可选文件，用于指定不希望被纳入版本控制的文件和目录。与.gitignore类似，但作用范围只限于本地仓库。

objects：这个文件夹是Git对象数据库的核心（上文提到的数据结构）。它包含了所有的对象，包括提交（commit）、树（tree）和文件（blob）对象等。

objects/info：这个目录包含了一些辅助索引文件，用于提高Git对象的查找性能

objects/pack：这个目录包含了Git对象数据库中的打包对象文件（pack files）。打包对象文件是一种高度压缩和优化的二进制文件，用于存储多个对象。它们以.pack为扩展名，并使用特定的压缩算法和索引结构。打包对象文件在Git中被用来节省磁盘空间和提高性能。

refs/heads：这个文件夹包含了本地分支的引用。每个分支都有一个相应的文件，其中包含指向分支最新提交的引用。

refs/tags：这个文件夹包含了标签（tag）的引用。每个标签都有一个相应的文件，其中包含指向标签的引用。

这些文件中主要需要留意的是objects和refs这两个文件夹。

前文提到的object都存储在objects文件夹中，其索引为hash，值为object的内容，git实现的方式是使用hash的前两位作为文件夹名称，后面的内容作为该文件夹下的文件名，文件的内容为object的内容。这样做的目的是便于组织文件和加快索引。

Git命令

基础

  • git help <command>: 获取帮助
  • git init: 创建git仓库
  • git status: 查看当前工作区状态
  • git add <filename>: 将文件添加到暂存区
  • git commit: 提交
  • git log: 查看提交历史
  • git log --all --graph --decorate: 以有向无环图的形式查看提交历史
  • git diff <filename>: 查看文件与暂存区中对应记录之间的差别
  • git diff <revision> <filename>: 查看文件与某次提交之间的差别，revision可以是reference如HEAD，也可以是commit的hash值
  • git checkout <revision>: 将HEAD与当前分支的head切换到历史的某次提交

分支与合并

  • git branch: 查看当前所有的分支
  • git branch <name>: 创建一个分支
  • git checkout -b <name>: 创建一个分支并切换到该分支
  • git merge <revision>: 将对应的提交的内容合并到当前的分支
  • git mergetool: 使用工具来解决合并冲突
  • git rebase: rebase set of patches onto a new base

远程仓库

  • git remote: 查看远端仓库

  • git remote add <name> <url>: 添加一个远端仓库

  • git push <remote> <local branch>:<remote branch>: 将本地的分支推送到远端，并更新远端的分支

  • git branch --set-upstream-to=<remote>/<remote branch>: set up correspondence between local and remote branch

    git branch –set-upstream-to=<remote>/<remote branch> 命令用于设置当前分支与远程分支之间的追踪关系，也称为 upstream。
    具体来说，–set-upstream-to=<remote>/<remote branch> 参数用于将当前分支与指定的远程分支进行关联。这样，Git 将会跟踪当前分支与远程分支之间的差异，使得后续的 Git 操作可以更加方便和自动化，例如 git push 和 git pull。
    设置 upstream 之后，可以通过以下方式获得一些好处：

    1. 默认推送和拉取：一旦设置了 upstream，你可以直接使用 git push 和 git pull 命令，而不需要指定远程分支的名称和位置。Git 会自动推送和拉取与当前分支关联的远程分支。
    2. 显示状态信息：使用 git status 命令时，Git 会显示当前分支与 upstream 分支之间的差异，告诉你当前分支落后于远程分支的提交。
    3. 更好的协作：当你与其他开发者共享远程仓库时，设置 upstream 可以更好地追踪彼此的进展，并确保及时更新和合并代码。

    在使用 –set-upstream-to 参数时，<remote> 是指远程仓库的名称（通常是 “origin”），<remote branch> 是指远程分支的名称。例如，origin/master 表示远程仓库 “origin” 的 “master” 分支。
    总结来说，git branch –set-upstream-to=<remote>/<remote branch> 命令用于将当前分支与指定的远程分支进行关联，使得 Git 可以跟踪差异、自动化推送和拉取，并提供更好的协作和状态信息。

  • git fetch: 从远端获取objects和reference

  • git pull: 等同于

    • git fetch
    • git merge

  • git clone: 下载远程仓库

撤销

  • git commit --amend: 将修改合并到上一次提交
  • git reset HEAD <file>: 放弃对某个文件的stage（暂存），即将该文件的状态变为当前提交的状态，但不会放弃该文件的修改，只是将其从暂存区中去除
  • git checkout -- <file>: 放弃对某个文件的修改

高级

  • git config: 修改git的配置
  • git clone --depth=1: 只克隆最新的一次提交
  • git add -p: 交互式的页面查看每一次提交，选择是否stage
  • git rebase -i: interactive rebasing
  • git blame: 查看每一行内容的修改者
  • git stash: temporarily remove modifications to working directory
  • git bisect: binary search history (e.g. for regressions)
  • .gitignore: 声明不需要记录的文件

Stash

本节内容来自chatGPT

stash就像一个栈一样存储内容，不与特定的分支绑定吗?

是的，你可以将 Git 中的 stash 视为一个栈（stack），其中可以存储多个储藏项。stash 不与特定的分支绑定，它是全局的，可在任何分支上应用和管理。
当你执行 git stash save 或简单地 git stash 命令时，Git 会将当前的工作目录和暂存区的修改保存到一个新的 stash 中，并将其添加到 stash 栈的顶部。每个 stash 都有一个唯一的标识符（stash@{n}），其中 n 是一个整数，表示 stash 的索引号，最新的 stash 具有索引号 0。
当你执行 git stash apply 或 git stash pop 命令时，Git 会从 stash 栈中取出最新的 stash，并将其应用到当前的分支上。这意味着 stash 不会自动关联到特定的分支，你可以在任何分支上选择应用 stash。
此外，你可以使用 git stash list 命令来查看当前的 stash 列表，显示所有的 stash 标识符和对应的描述信息。通过 git stash drop 命令可以删除指定的 stash，从 stash 栈中移除它。
总结来说，stash 是一个栈结构，可以存储多个储藏项，每个储藏项都是一个保存工作目录和暂存区修改的快照。stash 不与特定的分支绑定，你可以在任何分支上应用、管理和移除 stash。
